## 一、审计发现的核心问题

### 1.1 两套计算路径不一致（架构缺陷）

| 路径 | 函数 | 用途 | 问题 |
|------|------|------|------|
| 向量化 | `_vectorized_compute()` | 方案浏览器筛选 | 使用 np.clip / np.ceil |
| 标量 | `calculate()` | F5 重算 / Excel 导出 | 使用 ceil_to / 无钳制 |

**后果**: 浏览器中筛选出的"合格"方案，导出 Excel 后因标量重算产生偏差，出现 ✗。

**修复**: 方案应用时直接将向量化结果存入节点缓存（`NodeResult`），导出时使用缓存而非重算。

### 1.2 约束名称不匹配（3 处）

| 模块 | config 中 | 代码中 | 影响 |
|------|----------|--------|------|
| chuchenchi | `有效水深 h2>=2.0` | `有效水深 h2 >= 2.0` | 硬约束高亮失效 |
| gaomidu | `固体通量 ≤ 150` | `固体通量 <= 150` | Unicode vs ASCII |
| kw_chenshachi | B_single 在 fixed | vectorized 从 grid 读 | 崩溃 |

### 1.3 公式 Bug（多模块）

| 模块 | Bug | 修复 |
|------|-----|------|
| UV ziwai | N_rows = D_UV×v×h×1000/denom (量纲错) | 改为 ceil(D_UV/dose_per_row) |
| 沉砂池 chenshachi | 砂斗检查只比较 V_cone，忽略圆柱段 | V_storage = V_cone + V_cyl |
| 沉砂池 chenshachi | 标量 B 未钳制到 [H_max, 2*H_max] | 对标向量化加钳制 |
| CASS cass | 同上 | 加钳制 |
| 初沉池 chuchenchi | 堰负荷公式用单侧堰 | 改为双侧堰 (×2) |
| 平流沉砂池 kw_chenshachi | 堰负荷用 Q_max 而非 Q_single | 修正分母 |
| cost_estimator | _rectangular_tank node_type 缺失 | 7 个 BOQItem 全补 nt |
| cost_estimator | 池壁高度用 H 而非 H-tf | 统一改为 H-tf |
| unit_prices | PIPE 定义两次 (2024 被 2019 覆盖) | 删除重复, 保留 2024 价 |
| pipe_network_cost | 泵站 grand_total 之后才估算 | 移到之前 |

## 四、硬约束全量对齐

全部 13 个模块的 `constraint_names` 与代码中 `add_check()` 的名称已精确对齐（含空格、标点）。方案浏览器中所有约束均为硬约束，任意一项不达标即过滤。

---

## 五、导出一致性修复

- 方案应用时: 向量化结果直接写入 `node._result` (CLEAN 态缓存)
- 导出时: 优先使用缓存，无缓存时回退标量计算
- 输出过滤: Excel 写入时跳过硬约束失败节点
