## 三、Bug #1: 细格栅水头损失计算

### 根因
`geshan.py` 第 78 行：`s_m = 0.01` — 栅条宽度硬编码为 10mm，未区分粗/细格栅。

### 影响
- 细格栅 b=1.5mm 时 s/b=6.67，ξ=30.35，h₁=2.79m（误差 ~10×）
- 无水头损失约束，不合理的值未被检测

### 修复
| 文件 | 修改内容 |
|------|---------|
| `src/models/geshan.py` | 新增 `_s_default` 类属性：粗格栅 10mm，细格栅 3mm；`calculate()` 和 `_vectorized_compute()` 使用 `get_param("s")` / `fixed["s"]`；新增 `result.add_check("水头损失 h1", h1 <= 0.3, ...)` |
| `src/models/discretization.py` | cugeshan/xigeshan 固定参数新增 `s`，约束新增 `h1_loss` |
| `mods/core/cugeshan/discretization.json` | 同上 |
| `mods/core/xigeshan/discretization.json` | 同上 |
| `src/models/solution_space.py` | CONSTRAINT_LIMITS 新增 `"水头损失 h1": "<= 0.3"` |

### 效果
| 参数 | 修复前 | 修复后 |

| s (细格栅) | 10mm | 3mm |

## 四、Bug #2: 污泥干化太阳能面积计算

### 根因
`wuni_ganhua.py` 第 117 行：`A_dry = m_water_evap / (q_evap * 24)` — 面积公式未区分热干化与太阳能干化。`q_evap` 在热干化模式下单位为 kgH₂O/(m²·h)，需要 ×24 转换为日蒸发量；太阳能模式下单位应为 kgH₂O/(m²·d)，不应再 ×24。

### 影响
- 太阳能干化面积被低估 24 倍（52m² vs 实际 ~1254m²）

### 修复
| 文件 | 修改内容 |
|------|---------|
| `src/models/wuni_ganhua.py` | `execute_sludge()`: 面积公式分支（thermal `/q*24`，solar `/q`）；约束分支（thermal 4-15 kg/(m²·h)，solar 5-15 kg/(m²·d)）；`_vectorized_compute()`: 提取 `method` 并分支 |

### 效果
| 参数 | 修复前 | 修复后 |
|------|--------|--------|
| A_dry (solar, q=10) | 52 m² | 1254 m² |

## 五、Bug #3: UnboundLocalError — s_mm 变量定义顺序错误

### 根因
`geshan.py` 第 70 行 `result.params` 中引用了 `s_mm`，但 `s_mm` 在第 82 行才定义（位于栅槽宽度计算段）。在 `calculate()` 中添加 `"s": s_mm` 到 result.params 时，未注意到 `s_mm` 的定义在使用点之后。

```python
# Bug (修复前):
result.params = {..., "s": s_mm}  # ← s_mm 未定义！UnboundLocalError
...
s_mm = self.get_param("s")        # ← 这里才定义

# 修复后:
s_mm = self.get_param("s")        # ← 提前到参数读取区
result.params = {..., "s": s_mm}  # ← 已定义
```

### 影响
- 用户首次点击「计算」时，DAG 执行器调用 `calculate()` → `UnboundLocalError` → 格栅节点失败
- 异常被 `execute()` 的 `try/except` 静默捕获，log 中无报错
- 单独点击格栅节点后重算可能使用了缓存或 fallback 路径，h₁ 显示为 0.00

### 修复
| 文件 | 修改内容 |
|------|---------|
| `src/models/geshan.py:67` | 将 `s_mm = self.get_param("s")` 从第 82 行移至第 67 行（参数读取区） |
| `src/models/geshan.py:82` | 删除重复的 `s_mm` 定义，仅保留 `s_m = s_mm / 1000.0` |

### 同类检查
对本次修改的所有代码文件进行了变量定义顺序审计：
| 文件 | 结果 |
|------|------|
| `src/models/geshan.py` — `calculate()` | ✅ 已修复 |
| `src/models/geshan.py` — `_vectorized_compute()` | ✅ 无问题（`s_mm` 在 dtype/result 使用前已定义） |
| `src/models/wuni_ganhua.py` — `execute_sludge()` | ✅ 无问题（`method`、`A_dry`、`heat_total` 均在引用前定义） |

## 七、Bug #4: 水量输入机制重构 — Excel 解耦 + Kz 滑块 + 日流量重算

### 根因
1. Q_design 来自 Excel 读取，每个项目的 Excel 格式不同导致不可靠
2. `Q_design_override` 滑块改变 Q_design 后，`Q_avg_daily` 不重算（仍用 Excel 旧值）
3. Kz=1.4 硬编码，无法由用户调整
4. NaN 传播：Excel 异常时日流量永久损坏

### 修复
| 文件 | 修改内容 |
|------|---------|
| `src/models/pipe_network.py` | 移除 `Q_design_override`；新增 `Kz`（首位）和 `Q_design` 滑块参数；`calculate()`/`execute()` 每次重算 `Q_avg_daily = Q_design / Kz × 86400`；Excel 仅读取管道长度统计；`from_dict` 向后兼容旧项目 |
| `src/ui/main_window.py` | Scale 控件新增 `variable=var`，输入框 Enter 后滑块自动同步 |

### 效果
| 操作 | 结果 |
|------|------|
| 默认 Q=0.57, Kz=1.4 | Q_avg = 35177 m³/d |
| 改 Kz=2.0 | Q_avg = 24624 m³/d（即时重算） |
| 改 Q=1.0, Kz=2.0 | Q_avg = 43200 m³/d（即时重算） |
